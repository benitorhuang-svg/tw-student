# 功能規格書：SQLite 資料庫驅動的可觀測縣市 Atlas

**功能分支**: `003-local-db-prefetch-atlas`  
**建立日期**: 2026-03-07  
**狀態**: 實作中  
**輸入來源**: 使用者需求「直接建立 SQLite 資料庫、將官方 CSV 匯入資料庫、前端透過資料庫互動、維持繁體中文資料、所有超過 300 行的大檔需拆分並符合 SOLID」

## 使用者故事與測試情境

### 使用者故事一：只在高意圖情境預抓縣市細節（優先級 P1）

作為分析使用者，我希望系統只在我 hover 某縣市，或該縣市已經進入排行前幾名時才預先載入細節，讓畫面維持輕量，但我真的要點進去時又能更快回應。

**獨立驗證方式**：首頁進站後確認不會載入全部縣市細節；滑過某縣市或觀察排行前幾名後，再查看 Network 與介面觀測區，確認只補載對應縣市切片。

**驗收情境**：

1. **當** 使用者僅停留在全台首頁時，**則** 系統不得同步下載所有縣市細節。
2. **當** 使用者 hover 某個縣市，**則** 系統可在背景預抓該縣市細節與必要鄉鎮切片。
3. **當** 某縣市進入當前排行前幾名時，**則** 系統可在背景預抓其細節，但仍不得擴大成全量下載。

### 使用者故事二：以 SQLite 資料庫提供主互動資料層（優先級 P1）

作為重複切換縣市的使用者，我希望前端直接從 SQLite 資料庫查詢縣市、鄉鎮、學校與年度資料，而不是每次再依賴大量 JSON 檔，讓互動更穩定、查詢更一致。

**獨立驗證方式**：進站後先載入 SQLite 檔案，之後切換縣市與年份時確認畫面資料來自資料庫查詢，且觀測區能顯示 SQLite 命中。

**驗收情境**：

1. **當** 前端啟動時，**則** 系統必須先載入單一 SQLite 資料庫檔，作為縣市摘要、學校清單與年度趨勢的主來源。
2. **當** 使用者切換縣市、鄉鎮、學年度、學制或公私立條件時，**則** 系統必須透過資料庫查詢回傳對應資料，而不是重新抓取多個教育 JSON 切片。
3. **當** SQLite 檔尚未完成載入時，**則** 系統需顯示明確的載入狀態，不得出現白畫面或空資料誤判。

### 使用者故事三：學校清單升級為可排序表格（優先級 P1）

作為研究者，我希望縣市與鄉鎮的學校清單改成真正的資料表格，讓我能按學生數、年增減、狀態與其他欄位排序，並匯出目前的分析結果。

**獨立驗證方式**：點進任一縣市後確認學校區以表格呈現，點擊欄位標頭可排序，並能匯出 CSV。

**驗收情境**：

1. **當** 使用者進入縣市或鄉鎮模式後，**則** 學校區必須顯示可排序表格，而不是固定順序清單。
2. **當** 使用者切換排序欄位時，**則** 表格順序必須即時更新。
3. **當** 使用者點擊匯出時，**則** 系統必須匯出目前篩選結果的 CSV。

### 使用者故事四：把載入來源與資料庫大小視覺化（優先級 P2）

作為進階使用者，我希望能直接看到目前已載入哪些縣市切片、快取命中次數與檔案大小，讓我理解系統正在使用哪種載入來源。

**獨立驗證方式**：開啟首頁與切換縣市後，確認地圖與摘要區都能顯示來源觀測資訊。

**驗收情境**：

1. **當** 系統載入 SQLite 或邊界切片後，**則** 地圖或摘要區必須更新已載入資料來源狀態。
2. **當** 查詢命中記憶體快取或 SQLite 結果時，**則** 觀測區必須增加對應命中計數。
3. **當** 顯示資料來源時，**則** UI 必須能看到 SQLite 檔與邊界資產的大小。

### 使用者故事五：大型檔案必須原子化拆分（優先級 P1）

作為維護者，我希望前端與資料刷新程式都遵守檔案原子化原則，避免單一檔案承擔過多責任，讓後續資料庫查詢、地圖互動與分析計算都能分別測試與維護。

**獨立驗證方式**：檢查專案內主要原始碼檔案，確認超過 300 行的檔案已拆成多個單一責任模組，且整體 lint/build/test 仍通過。

**驗收情境**：

1. **當** 任一原始碼檔案超過 300 行且同時負責多種邏輯時，**則** 必須拆分為多個模組。
2. **當** 進行拆分後，**則** 各模組責任需符合單一職責與依賴反轉原則，不得形成新的巨型工具檔。
3. **當** 完成拆分後，**則** 原本公開 API 與使用者可見行為必須維持一致或經規格明確說明。

## 功能需求

- **FR-001**：資料刷新腳本必須將官方 CSV、點位與邊界整合後輸出為 SQLite 資料庫檔，內容需維持繁體中文。
- **FR-002**：前端必須以 SQLite 作為縣市摘要、鄉鎮摘要、學校明細、年度序列與 bucket 分群的主互動資料來源。
- **FR-003**：邊界圖資可維持 TopoJSON 切片，但教育資料互動不得再依賴 `counties/*.json` 作為主來源。
- **FR-004**：系統必須避免同一 SQLite 檔被重複併發載入，並提供記憶體層快取。
- **FR-005**：學校清單必須改為可排序表格，至少支援學生數、年增減、狀態、鄉鎮、學制、公私立排序。
- **FR-006**：學校表格必須提供 CSV 匯出能力。
- **FR-007**：地圖與摘要區必須顯示 SQLite 命中次數、邊界資產命中次數、累積傳輸量與檔案大小。
- **FR-008**：所有原始碼檔案若超過 300 行且職責過多，必須拆分成符合 SOLID 的原子模組。
- **FR-009**：版型必須延續 `local_view.html` 的控制台方向，強化地圖舞台、圖例與來源觀測區。

## 成功指標

- **SC-001**：首頁只需下載單一 SQLite 教育資料檔與必要邊界圖資，不再逐縣抓取教育 JSON。
- **SC-002**：重複切換縣市、鄉鎮與年份時，可在觀測區看到 SQLite 命中增加。
- **SC-003**：台中等大型縣市切換到學校工作台時，必須能顯示完整國小、國中、高中職與大專院校資料。
- **SC-004**：學校區能完成排序與 CSV 匯出。
- **SC-005**：地圖與摘要面板都能顯示載入來源觀測資訊。
