# 功能規格書：縣市細節串流與版型深化

**功能分支**: `002-streamed-county-atlas`  
**建立日期**: 2026-03-07  
**狀態**: 草案已核定，待實作  
**輸入來源**: 使用者需求「新增 specs 002、將教育資料拆為全台摘要與縣市細節、加入 E2E、沿用 local_view.html 地圖版型基底，且目前不需要本地資料庫」

## 使用者故事與測試情境

### 使用者故事一：先看全台，再按需載入縣市細節（優先級 P1）

作為首次進站的使用者，我希望首頁只先載入全台摘要與地圖必要資料，讓頁面能更快進入可操作狀態，而不是一開始就下載所有縣市學校細節。

**優先原因**：這是本輪效能優化的核心，直接決定首次載入成本。

**獨立驗證方式**：開啟頁面後檢查 Network，確認初次只載入全台摘要、縣市邊界與必要靜態檔；點擊縣市後才出現對應細節檔請求。

**驗收情境**：

1. **當** 首頁以全台模式開啟時，**則** 系統必須只載入全台摘要檔與縣市邊界，不得同步載入所有縣市學校細節檔。
2. **當** 使用者點擊某縣市時，**則** 系統必須按需載入該縣市細節檔，並更新鄉鎮排行、學校清單與摘要區。
3. **當** 使用者再次回到全台後再切回同一縣市時，**則** 系統應優先重用已載入快取，避免重複下載相同細節檔。

---

### 使用者故事二：深連結可還原到縣市與鄉鎮細節（優先級 P1）

作為需要分享分析結果的研究人員，我希望帶有縣市、鄉鎮、學年度與篩選條件的網址在重新開啟時，能先還原全台摘要，再自動補齊對應縣市細節與鄉鎮界線。

**優先原因**：資料改為延遲載入後，深連結不能退化，否則分享能力會被破壞。

**獨立驗證方式**：以帶 `county`、`township`、`year`、`level` 的網址直接開頁，確認畫面最終回到對應範圍，且不出現錯誤空白狀態。

**驗收情境**：

1. **當** 使用者以帶縣市與鄉鎮參數的網址進站時，**則** 系統必須自動載入該縣市細節檔與對應鄉鎮界線切片。
2. **當** URL 指向不存在的縣市或鄉鎮時，**則** 系統必須安全回退到最近有效的上層範圍。
3. **當** 細節檔尚未載入完成時，**則** 系統必須顯示可理解的載入狀態，不得出現錯誤或殘留舊資料。

---

### 使用者故事三：異常註記在串流細節模式下仍可辨識（優先級 P2）

作為資料審閱者，我希望即使學校與鄉鎮細節改成延遲載入，停辦、缺年度、整併或改制等註記仍能在載入後清楚顯示，而不是因資料分檔被遺漏。

**優先原因**：資料透明度不能因效能優化而下降。

**獨立驗證方式**：切換到存在異常註記的縣市或學校，確認載入完成後能看到狀態標記與說明文字。

**驗收情境**：

1. **當** 某縣市細節檔包含範圍註記時，**則** 摘要區或細節區必須顯示相對應的 `dataNotes`。
2. **當** 某學校被標記為 `待確認`、`停辦` 或存在 `missingYears` 時，**則** 學校列與單校焦點都必須保留這些資訊。
3. **當** 資料尚未載入完成時，**則** 系統不得先顯示錯誤的正常狀態來誤導使用者。

---

### 使用者故事四：以 local_view 結構深化資訊版型（優先級 P2）

作為分析平台使用者，我希望畫面更接近資料控制台，而不是單純上下堆疊的卡片，讓左側控制、中央地圖、右側摘要與下方表格有更明確的版位層級。

**優先原因**：這能提升資訊密度與操作效率，符合使用者指定的視覺方向。

**獨立驗證方式**：開啟桌機版與行動版頁面，確認主畫面具備控制欄、地圖主舞台、摘要面板與資料列表的清楚分區，且互動流程不變。

**驗收情境**：

1. **當** 桌面版頁面開啟時，**則** 主要畫面必須呈現更接近控制台的資訊分區，而不是單一路徑卡片堆疊。
2. **當** 使用者切換縣市或鄉鎮時，**則** 地圖主舞台、摘要卡與排行模組必須維持一致的版位與狀態回饋。
3. **當** 行動版顯示時，**則** 控制與摘要模組必須可折行重排，維持完整可操作性。

## 邊界情境

- 深連結在細節檔尚未下載完成前，不得誤判為「無資料」。
- 某縣市細節檔載入失敗時，頁面必須只影響該縣市視圖，不能讓全台摘要一併失效。
- 同一縣市若被多次點擊，不得重複累加請求或造成載入狀態閃爍。
- JSON 產物拆分後，仍必須保持多行 prettified 格式，避免產生單行輸出。

## 功能需求

### 功能性需求

- **FR-001**：系統必須將教育資料拆分為全台摘要檔與縣市細節檔。
- **FR-002**：首頁初次載入時，系統不得同步下載所有縣市學校細節檔。
- **FR-003**：系統必須在選取縣市或透過深連結指定縣市時，按需載入對應縣市細節檔。
- **FR-004**：縣市細節檔必須包含鄉鎮摘要、學校清單、單校趨勢所需欄位與範圍註記。
- **FR-005**：URL 還原流程必須支援「先載摘要，再補縣市細節，再補鄉鎮切片」的多階段初始化。
- **FR-006**：系統必須提供明確的細節載入中狀態與錯誤訊息。
- **FR-007**：系統必須保留既有 `dataNotes`、`status`、`missingYears` 契約，不得因資料拆分而遺失。
- **FR-008**：系統必須加入至少三條 E2E 測試，覆蓋 URL 還原、縣市切換載入鄉鎮切片、異常註記顯示。
- **FR-009**：系統必須沿用 `local_view.html` 的資訊架構方向，重整控制區、地圖主舞台與資料表格區。
- **FR-010**：系統目前不得引入本地資料庫、IndexedDB 同步層、DuckDB-WASM 或額外 API 服務作為主要解法。
- **FR-011**：所有新產生的 `.json` 檔與更新後的 `.topo.json` 檔都必須保持 prettified 多行格式。

### 關鍵實體

- **全台摘要檔**：首頁啟動所需的全台與縣市摘要資訊，包含篩選、地圖著色與預覽統計必要欄位。
- **縣市細節檔**：單一縣市的鄉鎮摘要、學校細節、資料註記與趨勢資料。
- **初始化狀態機**：描述 URL 還原時的摘要已載入、縣市細節已載入、鄉鎮切片已載入等階段。
- **E2E 驗證路徑**：以使用者流程觀點定義自動測試可觀察的 UI 與載入行為。

## 成功指標

### 可衡量成果

- **SC-001**：初次進站時，Network 中不會出現所有縣市細節檔同時下載。
- **SC-002**：帶有縣市與鄉鎮參數的深連結可在單次進站內還原到對應分析範圍。
- **SC-003**：至少三條 E2E 主流程可穩定通過。
- **SC-004**：畫面版型在桌機下呈現更清楚的控制台分區，且行動版仍可操作。
- **SC-005**：本輪文件與資料產物持續維持繁體中文與 prettified JSON 輸出規範。
