# 實作計畫：台灣學生人數互動探索平台

**分支**: `001-interactive-edu-map` | **日期**: 2026-03-07 | **對應規格**: `/specs/001-interactive-edu-map/spec.md`
**輸入來源**: `/specs/001-interactive-edu-map/spec.md` 的優化後需求

## 摘要

本輪實作要將既有正式資料互動儀表板推進到可分享、可驗證、可持續擴充的狀態。重點包含：

1. 將分析狀態同步到 URL，使縣市、鄉鎮、學年度與學制可被分享與直接還原。
2. 將大型地理資料與正式統計資料改為較輕量的前端載入策略，例如 TopoJSON 壓縮與資料切片。
3. 在學校與行政區層級加入正式資料註記，揭露停辦、缺年度與改制差異。
4. 依照 `local_view.html` 所示方向修正地圖視覺，強化圖例、分級著色與左側控制區的一致性。
5. 建立專案專用 skill，將本專案的資料處理、地圖設計、SDD 流程與驗證規則沉澱為可重複使用的工作流資產。

## 技術背景

**語言 / 版本**: TypeScript 5.9、React 19  
**主要依賴**: React、Vite、d3-geo、csv-parse、shpjs  
**資料層**: 由資料刷新腳本產出的靜態正式資料檔，並在前端進行分段載入與 URL 還原  
**測試方式**: ESLint、TypeScript build、手動 quickstart 驗證  
**目標平台**: 現代桌面與行動瀏覽器  
**專案型態**: 單頁前端應用  
**效能目標**: 正式資料導入後，首次可互動內容維持在可接受等待範圍內，篩選與層級切換維持主觀即時反應  
**限制條件**: 單頁面下鑽、繁體中文、優先使用靜態資料產物，不預設引入本地資料庫或後端服務  
**目前範圍**: 全台、縣市、鄉鎮市區、學校層級正式資料探索與分享

## 架構決策

- 仍維持單一前端應用，因為目前需求重點是互動探索與靜態資料交付，而非多使用者即時查詢。
- 地理資料優先改成 TopoJSON 或等效壓縮產物，再於前端還原為可繪製資料，以降低邊界檔體積。
- 正式教育資料將拆分為「全台摘要層」、「縣市層摘要」、「學校細節層」等切片，避免每次進站都讀全量深層資料。
- URL 狀態同步由前端控制，不需要為此引入路由框架或後端 session。
- 本地端資料庫暫不作為本輪主要方案，因為目前讀取模式較偏靜態報表與切片檔載入，先用靜態產物與瀏覽器端快取即可達成更低維運成本的效果。

## 複雜度追蹤

| 項目 | 為何需要 | 為何不採更簡單方案 |
|------|----------|--------------------|
| TopoJSON / 切片資料 | 正式資料量已明顯超過示範資料，需降低首次載入成本 | 若繼續直接載入完整 GeoJSON 與全量學校資料，效能會退化 |
| URL 狀態同步 | 需支援分享、重整還原、深連結 | 只用 local state 無法分享分析上下文 |
| 資料註記模型 | 正式資料存在缺年、停辦與改制差異 | 若不建模，只能在 UI 臨時硬寫，後續難維護 |
