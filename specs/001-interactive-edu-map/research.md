# 研究紀錄：台灣學生人數互動探索平台

## 決策一：維持 React + Vite + TypeScript 前端單頁架構

- **決策**：維持以 Vite 為基礎的 React + TypeScript SPA。
- **原因**：需求主體仍是高互動前端分析介面，URL 狀態同步、正式資料載入、切片與地圖下鑽都可在單頁架構中完成，無須為此提前導入伺服器端框架。
- **備選方案**：
  - Next.js：目前不需要 SSR、路由檔案系統或伺服器動作，導入成本高於收益。
  - 純原生 JavaScript：當互動狀態與資料切片增加後，可維護性不足。

## 決策二：真實行政區輪廓優先採 TopoJSON 或等效壓縮產物

- **決策**：正式地理資料輸出優先改為壓縮格式，前端再還原為繪圖資料。
- **原因**：縣市與鄉鎮邊界資料在正式版已具規模，若直接傳送完整 GeoJSON，首次載入體積與解析成本都偏高。
- **備選方案**：
  - 繼續直接載入完整 GeoJSON：實作較簡單，但對正式資料場景效能較差。
  - 使用地圖瓦片服務：會引入額外服務依賴，超出目前交付邊界。

## 決策三：正式教育資料改為分層切片，不一次載入全量學校細節

- **決策**：資料刷新腳本輸出至少包含全台摘要、縣市細節與地理邊界等切片檔。
- **原因**：首頁其實只需要總覽與地圖著色資料，沒有必要每次進站就下載所有學校細節。
- **備選方案**：
  - 全量單一 JSON：結構直觀，但體積與解析時間較差。
  - 導入本地資料庫：目前查詢模式偏靜態，維運成本高於收益。

## 決策四：分享連結使用 URL 查詢參數，而非額外狀態儲存服務

- **決策**：以 URL 查詢參數保存學年度、學制、縣市、鄉鎮、區域與搜尋字串。
- **原因**：使用者最直接的需求是可複製、可分享、可重整還原，URL 是最低成本且最可預期的解法。
- **備選方案**：
  - localStorage：可保留狀態，但不利分享與跨裝置重現。
  - 後端短網址服務：超出目前需求範圍。

## 決策五：正式資料異常以結構化註記模型處理

- **決策**：將停辦、缺年度、名稱異動、行政區改制差異等資訊整理為獨立的資料註記欄位。
- **原因**：若只在 UI 臨時判斷，後續很難維護，也不利於資料產線與前端共享邏輯。
- **備選方案**：
  - 在 UI 端零散判斷：容易重複邏輯且難以驗證。
  - 完全忽略異常資訊：會削弱正式資料的可信度。

## 決策六：目前不建立本地端資料庫，先採靜態切片加瀏覽器快取

- **決策**：本輪不新增 SQLite、DuckDB 或其他本地資料庫層。
- **原因**：目前資料是批次刷新、靜態讀取、前端展示為主，使用預切片 JSON / TopoJSON 加上瀏覽器快取即可明顯改善讀取體驗，且部署與維護成本更低。
- **何時再考慮資料庫**：若後續出現自由查詢、跨年份複雜聚合、模糊搜尋索引或資料量大到靜態產物難以管理時，再評估 IndexedDB、DuckDB-WASM 或本地 API 層。

## 決策七：地圖視覺參考 `local_view.html` 的資訊密度與控制區配置

- **決策**：吸收參考畫面中的左側控制面板、分級圖例與地圖著色層次，但仍保留目前專案的品牌與互動結構。
- **原因**：目前地圖雖已改為真實輪廓，但視覺層次與資訊指引仍可更聚焦，尤其是圖例、主標題與快速統計區。
- **備選方案**：
  - 完全照抄參考頁：不利維持本專案既有互動與版面邏輯。
  - 完全不參考：會錯失更成熟的地圖控制與資訊密度配置。
