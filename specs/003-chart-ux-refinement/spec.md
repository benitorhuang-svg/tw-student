# 功能規格書：圖表與 UX 互動精修

**功能分支**: `003-chart-ux-refinement`  
**建立日期**: 2026-03-11  
**狀態**: Proposed  
**輸入**: 使用者需求：「圖表與 UX 互動精修」

## User Scenarios & Testing *(mandatory)*

### 使用者故事 1 - 補齊決策圖表缺口 (Priority: P1)

教育分析使用者在概況總覽、區域分析與單校聚焦頁面，必須能直接看到對應產品意圖的核心圖表，而不是以近似圖表或文字說明替代。

**Why this priority**: Treemap、Butterfly Chart、PR Indicator 與學校分布分析是產品規格已明確期待但目前缺漏的核心洞察入口；若未補齊，圖表體驗仍無法對應產品承諾。

**Independent Test**: 分別進入概況總覽、區域分析、單校聚焦與學校分布分析相關區塊，確認各頁面皆能獨立顯示對應圖表、空狀態與互動說明，且不依賴其他頁面才能理解。

**Acceptance Scenarios**:

1. **Given** 使用者停留在概況總覽頁，**When** 檢視地區量體模組，**Then** 系統顯示以區域為第一層、縣市為第二層的矩形圖，並以學生量體呈現階層差異。
2. **Given** 使用者停留在區域分析頁，**When** 檢視公私立結構模組，**Then** 系統顯示左右對稱的蝴蝶圖，讓使用者可直接比較各縣市或區域的公私立比例差異。
3. **Given** 使用者已聚焦單一學校，**When** 檢視定位指標模組，**Then** 系統顯示該校在同縣市、同學制可比 cohort 中的百分等級指標與相對定位說明。
4. **Given** 使用者查看學校規模分布分析，**When** 資料量足以呈現分布，**Then** 系統以真正的直方圖作為主要分布圖表，而非僅以盒鬚圖代理。

---

### 使用者故事 2 - 統一圖表互動與視覺品質 (Priority: P1)

教育分析使用者在不同分頁來回切換時，應感受到一致的圖表語言，包括色彩、圖例、提示資訊、動畫節奏、響應式行為與鍵盤焦點回饋，而不是每張圖各自為政。

**Why this priority**: 目前圖表存在色彩 token 使用不一致、tooltip 風格分裂、動態效果採用不一、響應式 SVG 表現不足與可及性缺口，直接影響專業儀表板品質。

**Independent Test**: 在桌機與手機檢視所有目標圖表，逐一驗證 tooltip、legend、動態、焦點樣式、縮放與版面適應是否一致，且不需依賴特定瀏覽方式才能使用。

**Acceptance Scenarios**:

1. **Given** 使用者在任一目標圖表上使用滑鼠、觸控或鍵盤聚焦資料項，**When** 互動狀態改變，**Then** tooltip、highlight 與焦點樣式遵循相同的視覺規則與資訊層級。
2. **Given** 使用者在手機寬度與桌機寬度間切換，**When** 重新檢視相同圖表模組，**Then** 圖表內容不應被裁切、重疊或要求水平捲動才能理解主要資訊。
3. **Given** 使用者偏好減少動態效果，**When** 進入圖表頁面，**Then** 系統保留資訊可讀性，但避免依賴強制動畫才能理解狀態變化。

---

### 使用者故事 3 - 降低分析面板複雜度以維持一致性 (Priority: P2)

產品維護者在擴充新圖表與 UX 規則時，需要用一致的面板編排與設定方式管理圖表區段，避免單一畫布與分頁面板承擔過多零散責任，導致同類互動難以同步更新。

**Why this priority**: 目前 DashboardCanvas 與多個 tab panel 的責任邊界偏大，prop surface 過寬；若不先整理圖表區段編排，後續新圖表與 UX 統一規則會持續分散。

**Independent Test**: 檢視目標面板是否能以一致的區段定義方式組成主要圖表模組，並確認新增或調整圖表互動規則時不需在多處重複描述相同配置。

**Acceptance Scenarios**:

1. **Given** 需要在現有分析頁面加入或調整圖表區段，**When** 維護者更新頁面定義，**Then** 圖表標題、說明、互動規則與顯示條件可在集中且可預期的位置維護。
2. **Given** 同一類圖表互動規則需要調整，**When** 維護者套用新規範，**Then** Overview、Regional、County 與 School Focus 相關面板不需各自複製不一致邏輯。

---

### 使用者故事 4 - 建立桌機與手機視覺驗證門檻 (Priority: P2)

產品團隊在交付圖表精修功能前，需要明確的桌機與手機視覺驗證準則，確保這次精修不是只在單一尺寸看起來正常。

**Why this priority**: 本專案已明確朝全視窗 PWA 體驗優化，圖表若僅在桌機成立，將直接削弱行動裝置的專業感與可用性。

**Independent Test**: 依規格執行桌機與手機視覺檢視，確認所有目標圖表模組都通過版面、互動與資訊呈現驗證。

**Acceptance Scenarios**:

1. **Given** 功能即將交付，**When** 團隊執行桌機視覺檢查，**Then** 目標圖表模組在桌機寬度下具有完整資訊階層、標示清晰與互動一致性。
2. **Given** 功能即將交付，**When** 團隊執行手機視覺檢查，**Then** 目標圖表模組在手機寬度下維持可讀標示、可觸達互動與無明顯排版破損。

### Edge Cases

- 當 `/data/*` 資產路徑錯誤、漏部署或被 SPA fallback 攔截時，系統需辨識「伺服器回傳 HTML 而非正式資料」，並提供可理解的錯誤訊息，而不是直接顯示原始 JSON parse 例外。
- 當 Treemap 某層級只有單一節點或多個節點學生數接近零時，系統仍需提供可辨識標示，不能因面積過小而完全失去資訊。
- 當 Butterfly Chart 一側資料為零或該地區僅有單一管理別資料時，系統需明確顯示失衡或缺值狀態，而非繪製誤導性的對稱圖形。
- 當 PR Indicator 的比較 cohort 樣本不足、資料缺漏或無法形成合理百分等級時，系統需改以不可計算狀態與原因說明呈現。
- 當學校數過少無法形成有意義的直方圖分箱時，系統需降級為簡化分布摘要，並說明為何未顯示完整分布圖。
- 當 tooltip 依賴 hover 的情境出現在觸控裝置上，系統仍需提供等價的點擊或聚焦資訊揭露方式。
- 當使用者採用鍵盤或輔助工具瀏覽，圖表主要互動點必須具有可見焦點與可理解描述，不能只靠顏色變化傳達狀態。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 在概況總覽頁新增矩形圖，以區域與縣市階層呈現學生量體，並保留與既有 atlas 視覺語言一致的資訊層級。
- **FR-002**: 系統 MUST 在區域分析頁新增真正的蝴蝶圖，用於比較公私立學生結構，且不得以一般長條圖或堆疊比例條替代主要分析角色。
- **FR-003**: 系統 MUST 在單校聚焦頁新增 PR Indicator，顯示該校在同縣市、同學制可比 cohort 內的百分等級定位與相對解讀。
- **FR-004**: 系統 MUST 將學校分布分析的主要圖表策略明確定義為「新增真正的直方圖」；現有盒鬚圖不得再單獨作為直方圖代理，但可保留為補充摘要。
- **FR-005**: 系統 MUST 為目標圖表建立一致的 tooltip、legend、色彩 token 與狀態標示規則，使不同頁面的圖表互動語言一致。
- **FR-006**: 系統 MUST 統一圖表進場與狀態轉換的動態規則，讓目標圖表共享同一套動畫節奏與降級行為。
- **FR-007**: 系統 MUST 改善目標圖表的響應式 SVG 與容器行為，確保桌機與手機下皆能完整呈現主要資訊而不破版。
- **FR-008**: 系統 MUST 為目標圖表提供鍵盤操作、可見焦點、文字替代與非色彩依賴的狀態辨識，補足目前可及性缺口。
- **FR-009**: 系統 MUST 降低主要分析面板與分頁容器的配置複雜度，將重複的圖表區段定義與互動規則集中管理，避免擴充新圖表時持續放大 prop-drilling 與頁面責任混雜。
- **FR-010**: 系統 MUST 在交付前完成桌機與手機的視覺驗證，涵蓋至少概況總覽、區域分析、縣市分析與單校聚焦中受此次功能影響的圖表模組。
- **FR-011**: 系統 MUST 維持本功能為前端體驗精修範圍，不得要求新的部署架構、即時服務或後端依賴作為本功能成立前提。
- **FR-012**: 若本功能的圖表範圍、互動定義或驗證標準發生變更，團隊 MUST 先更新本規格文件，再進行對應程式實作。
- **FR-013**: 系統 MUST 統一正式資料資產的 URL 組裝規則，並在 `/data/*` 路徑錯誤時回傳可診斷的錯誤，而不是讓 HTML fallback 混入 JSON / SQLite 載入流程。

### Non-Functional Requirements

- **NFR-001**: 圖表精修成果 MUST 維持現有 Taiwan education atlas 的產品語言，不得演變成與地圖工作台脫節的通用型儀表板重設計。
- **NFR-002**: 目標圖表模組 MUST 具備專業儀表板等級的資訊階層，讓標題、摘要、圖例、資料重點與輔助說明能在單一視區內被快速理解。
- **NFR-003**: 色彩系統 MUST 以一致 token 為主，避免頁面或元件各自定義 fallback 色碼造成視覺漂移。
- **NFR-004**: 動畫表現 MUST 服務於資訊揭露與狀態轉換，不得造成閱讀延遲、過度閃動或讓重要資訊僅在動畫完成後才可辨識。
- **NFR-005**: 響應式行為 MUST 同時支援桌機與手機主要使用情境，並符合既有 PWA 版面優化方向。
- **NFR-006**: 本功能的調整 MUST 以重用既有資料來源與前端衍生資料能力為前提，不得擴張為資料管線、部署或平台治理改造專案。
- **NFR-007**: 正式資料載入失敗時，錯誤訊息 MUST 指向資產路徑 / 載入型別問題，避免將原始 parser 例外直接暴露為唯一診斷資訊。

### Key Entities *(include if feature involves data)*

- **圖表區段定義**: 代表分析面板中的單一圖表模組，包含標題、摘要、顯示條件、互動說明與驗證責任。
- **可比 Cohort**: 代表用於 PR Indicator 與分布分析的同縣市、同學制學校集合，是定位與比較的共同基準。
- **分布分箱**: 代表直方圖用來描述學校規模分布的區間集合，用於揭露集中、偏態與極端值現象。
- **圖表互動規則**: 代表 tooltip、legend、焦點、選取與高亮等互動語言的共用規範。
- **視覺驗證清單**: 代表桌機與手機交付前必須完成的圖表檢視項目，用於判定是否達到專業品質門檻。

## Assumptions

- 既有前端資料切片已具備區域、縣市、學校與管理別等基本維度，可支撐此次圖表新增與重組，不需要新增部署層級服務。
- PR Indicator 所需的比較基準以同縣市、同學制學校集合為主，除非後續規格修訂，否則不擴大為跨縣市全國排名。
- 直方圖的主要用途是揭露學校規模分布形狀；盒鬚圖若保留，僅作補充統計摘要，而非主要洞察介面。
- 此次精修以既有 atlas 視覺語言延伸為原則，保留目前地圖工作台、分析面板與分頁脈絡，不進行整體品牌重設計。
- 前端內部可自行選擇合適的組織方式降低面板複雜度，但不得改變對外部署邊界或引入新平台依賴。

## Out of Scope

- 變更 GitHub Pages、PWA、靜態切片或既有部署架構。
- 以新後端 workspace、即時 API 或額外服務取代目前前端資料驅動模式。
- 重新定義 atlas 的整體資訊架構、主導航或地圖下鑽模型。
- 與本次圖表精修無直接關聯的資料刷新腳本重構、行政區邊界策略調整或正式資料治理規格改寫。
- 將所有既有圖表全面重做為單一大型改版；本次以補齊缺口與統一規則為主。

## 實作邊界

- 本功能限定為 `frontend/` 使用者體驗與圖表模組精修，不得以需要新部署拓樸為前提。
- 允許整理前端面板組裝方式與重用規則，但不得擴張為跨 workspace 的平台遷移專案。
- 若後續發現資料不足以支撐某一圖表表達，應先回到本規格修訂需求與驗證方式，再決定是否調整實作範圍。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 概況總覽、區域分析與單校聚焦三個目標頁面，皆完成對應新圖表模組上線條件，且不存在以舊圖或文字暫代核心洞察的情況。
- **SC-002**: 受此次功能影響的目標圖表模組在桌機寬度與手機寬度的最終視覺檢查中，零容忍資訊裁切、標示重疊與橫向捲動缺陷。
- **SC-003**: 目標圖表模組的互動審查中，100% 主要互動點皆可由鍵盤或等價非滑鼠方式觸達，並具備可見焦點與文字化說明。
- **SC-004**: 目標圖表模組在交付前的設計一致性檢查中，tooltip、legend、色彩 token 與動態規則不得出現重大例外。
- **SC-005**: 維護者可在不變更部署架構的前提下，於既有前端工作台內完成本功能交付，且任何後續範圍變更都先反映於本規格文件。
